# Дизайн обращений

## Цель

Добавить логику обращений: любой пользователь VK может отправить обращение с необязательными картинками, а пользователи с ролью менеджера VK, указанной в `vkbot.editor-roles`, могут просматривать открытые обращения и отвечать на них текстом. После ответа редактора исходный пользователь получает ответ.

## Пользовательский сценарий

1. Пользователь нажимает кнопку "Обращение".
2. Бот просит отправить текст обращения и, при необходимости, прикрепить картинки к тому же сообщению.
3. Бот сохраняет обращение со статусом `OPEN`, возвращает стандартную клавиатуру и подтверждает, что обращение принято.
4. Редактор нажимает кнопку "Обращения".
5. Бот показывает открытые обращения кнопками с короткими id. Если открытых обращений нет, бот отправляет сообщение о пустом списке и возвращает стандартную клавиатуру редактора.
6. Редактор выбирает обращение.
7. Бот отправляет детали выбранного обращения: короткий id, VK id автора, текст, время создания и ссылки на картинки, если они есть. Затем бот просит редактора отправить текстовый ответ.
8. Редактор отправляет текст ответа.
9. Бот помечает обращение как `ANSWERED`, сохраняет данные ответа, возвращает клавиатуру редактора и отправляет текст ответа автору обращения.
10. На любом шаге создания обращения или ответа кнопка "Отмена" очищает текущее состояние и возвращает стандартную клавиатуру.

## Границы

Ответы на обращения только текстовые. Пользователи могут прикладывать картинки только при создании обращения. В первой версии сохраняются URL фотографий из callback attachments, без скачивания файлов, потому что редакторам достаточно открыть картинки по ссылкам, а VK уже передает подходящие URL в callback payload.

Для создания обращения обязателен непустой текст. Картинки необязательны. Если пользователь начал сценарий и отправил только фотографии без текста, бот попросит отправить текст еще раз и оставит пользователя в состоянии создания обращения.

## Архитектура

Используется существующий маршрут callback и команд:

`CallbackController -> CallbackService -> ICallbackHandler -> CommandHandlerService -> CommandHandler`

Добавляются:

- `CommandType.CREATE_APPEAL` и `CreateAppealCommandHandler` для начала создания обращения пользователем.
- `CommandType.APPEALS` и `AppealsCommandHandler` для показа открытых обращений редакторам.
- `CommandType.SELECT_APPEAL` и `SelectAppealCommandHandler` для выбора обращения, на которое редактор будет отвечать.
- `AppealInputHandler` для входящего текста обращения и текста ответа редактора.
- `Appeal`, `AppealStatus`, `AppealRepository` и `AppealService` в стиле существующих model и utils.
- `vkbot.appeals.storage.path` для JSON-хранилища по аналогии с `vkbot.events.storage.path`.

В `UserState` добавляются:

- `AWAITING_APPEAL_TEXT`
- `AWAITING_APPEAL_ANSWER`

Для `AWAITING_APPEAL_ANSWER` добавляется небольшой in-memory `AppealAnswerDraftCache`, где ключом будет VK id редактора, а значением выбранный id обращения. Это повторяет подход `EventDraftCache` и оставляет `UserStateCache` универсальным.

## Хранение данных

Обращения хранятся JSON-массивом в файле из конфигурации. Каждое обращение содержит:

- `id`
- `userId`
- `text`
- `photoUrls`
- `status`
- `createdAt`
- `answeredAt`
- `answeredByUserId`
- `answerText`

`AppealRepository` отвечает только за файловый IO и сериализацию. `AppealService` отвечает за генерацию id, переходы статусов, фильтрацию открытых обращений и валидацию, например непустой текст обращения и непустой текст ответа.

## Маршрутизация

`CallbackService.defineType` направляет `message_new` в `appeal_input`, когда состояние пользователя равно `AWAITING_APPEAL_TEXT` или `AWAITING_APPEAL_ANSWER`. Существующая специальная обработка `cancel` остается первой, чтобы отмена работала из обоих состояний обращений.

Сообщения с фото не должны автоматически попадать в `uploadPhotoHandler`, когда пользователь создает обращение. `CallbackService` должен проверять активное состояние пользователя до универсального маршрута `upload_photo`, чтобы сообщения обращений с фотографиями обрабатывал `AppealInputHandler`.

## Клавиатура

В стандартной клавиатуре появляется кнопка "Обращение" для всех пользователей. В редакторском блоке появляется кнопка "Обращения".

Добавляется клавиатура списка обращений, которая выводит id открытых обращений по строкам и включает кнопку "Отмена". Выбор кнопки отправляет payload:

`{"command": "select_appeal", "appeal_id": "<id>"}`

В `Payload` добавляется поле `appealId`.

## Права доступа

Список обращений и выбор обращения доступны только пользователям с ролью редактора. `AppealsCommandHandler` и `SelectAppealCommandHandler` оба проверяют `RoleIdentifier.hasEditorRights(...)`. Если не-редактор вручную вызовет эти команды, бот откажет и вернет обычную клавиатуру.

Создавать обращение может любой пользователь.

## Обработка ошибок

Если редактор выбирает обращение, которого уже нет или на которое уже ответили, бот отправляет понятное сообщение и снова показывает актуальный список открытых обращений.

Если отправка финального ответа исходному пользователю падает через VK API, handler должен пробросить ошибку как runtime exception, как это делают существующие обработчики. Обращение должно помечаться отвеченным только после успешной отправки исходному пользователю, чтобы обращение не потерялось без уведомления.

## Тестирование

Добавить сфокусированные тесты:

- `AppealService` создает обращения, показывает только открытые обращения и помечает обращение отвеченным.
- `CallbackService` направляет состояния обращений в `appeal_input`, включая сообщения с фото.
- `StandardKeyboard` показывает "Обращение" всем пользователям и "Обращения" только редакторам.
- `CancelCommandHandler` очищает данные черновика ответа на обращение.
- `AppealInputHandler` валидирует пустой текст пользователя, сохраняет текст обращения с URL фотографий, валидирует пустой ответ редактора, отправляет ответ исходному пользователю и помечает обращение отвеченным после успешной отправки.
